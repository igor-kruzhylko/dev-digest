## What Doesn't Work
<!-- dead ends & antipatterns to avoid (highest-value section — don't skip it) -->
- 2026-06-29 — Do NOT add a REQUIRED field (or a `z.array(...).default([])`, whose zod OUTPUT type is required) to `PrMeta`: it is reused as the return type of the GitHub adapter (`listPullRequests`) and as the base of `PrDetail` (`getPullRequest` + the offline fallback in `pulls/routes.ts`), none of which can supply list-only data. Typecheck then breaks in `adapters/github/octokit.ts`, `adapters/mocks.ts`, and the PR-detail handler. List-endpoint-only fields must be optional — use `.nullish()` (matches `score`/`cost_usd`); the route fills it, everyone else omits it (server/src/vendor/shared/contracts/platform.ts:176). Remember to mirror the edit into BOTH vendored copies (server + client).
- 2026-06-25 — Do NOT derive a parent dir with forward-slash string math (`full.lastIndexOf('/')` / `full.slice(0, slash)`) on a `path.join()` result — on Windows `join` emits backslashes, so the slash index is -1, the `mkdir` guard never fires, and `writeFile` then ENOENTs. Bit the `writeFileAt` test helper, failing 6 `runIncremental` cases on Windows only (green on POSIX). Use `mkdir(dirname(full), { recursive: true })` (node:path is separator-agnostic; `recursive` is a no-op if it exists) — which is the fix now in place (server/test/indexer-pipeline.test.ts:140).
- 2026-06-21 — Do NOT use `` import.meta.url === `file://${process.argv[1]}` `` for CLI entrypoint detection — on Windows the URL forms never match, so `pnpm db:migrate` / `pnpm db:seed` exit 0 without running. Use `isDirectRun(import.meta.url)` from `server/src/db/is-direct-run.ts` (`pathToFileURL`); the caller must pass its own `import.meta.url`, not rely on the helper's.

## Codebase Patterns
<!-- conventions & architectural choices specific to this module -->
- 2026-07-02 — A module that must parse UNTRUSTED uploaded content (the skills import — `.md`/`.zip` via a hand-rolled frontmatter reader + `fflate`) gets a fifth file alongside the usual `routes/service/repository/helpers/constants` triple: `import.ts`, a PURE function (`previewImport`) with zero DB/I/O beyond the in-memory unzip. It is called only from `service.ts`, never DB-writes, and every non-Markdown archive entry is listed in `ignored_files` without its content ever being read past what `fflate.unzipSync` needs to enumerate it. The preview→confirm split (this endpoint returns a DTO; a SEPARATE `POST /skills` persists it) is the actual trust boundary — not runtime sandboxing. Mirror this file shape for any future "import from an untrusted upload" feature (server/src/modules/skills/import.ts).
- 2026-06-30 — Shared HTTP contracts belong in @devdigest/shared, then in BOTH vendored copies: agent bodies/params and repo-intel state now share one Zod source across server routes and client hooks. Map server-only fields (for example Date) to transport strings at the route boundary, not in the hook (server/src/modules/repo-intel/routes.ts:22, server/src/vendor/shared/contracts/platform.ts:280).
- 2026-06-29 — `ReviewService.deleteReview` deletes the WHOLE agent run when the review has a `run_id` (delegates to `deleteAgentRun`), not just the review row — otherwise the timeline run would linger with no findings while the list/accordion lose them, leaving the two surfaces inconsistent. It first loads the review and bails (`false`) if missing or cross-workspace. `deleteAgentRun` returns `true` if EITHER the reviews OR the `agent_runs` row was deleted (a review can outlive its run row), so callers don't 404 a successful partial cleanup (server/src/modules/reviews/service.ts:60, server/src/modules/reviews/repository/run.repo.ts:80).
- 2026-06-29 — `GET /repos/:id/pulls` now also returns `PrMeta.findings` (ALL findings across the PR's `kind='review'` runs), computed on read like `score`/`cost_usd` — one `findings ⨝ reviews` IN-query grouped per `pr_id` in JS, no denorm. Mirrors the detail page's flatten-across-runs. It's NOT filtered by accepted/dismissed (the list shows everything). Add list-only aggregates here, not as stored columns (server/src/modules/pulls/routes.ts:147).
- 2026-06-25 — Cross-cutting repositories are exposed on the DI Container, never imported across modules. Added `container.repoRepo` (RepoRepository) + `container.pullsRepo` (PullsRepository) next to the existing `agentsRepo`/`reviewRepo`; the `pull_requests`/`pr_files`/`pr_commits` writes for BOTH `pulls` and `polling` go through `container.pullsRepo`. Why: a module must not import a sibling's `repository.ts` (dependency-cruiser `no-cross-module-repository`). How to apply: need another module's table? expose its repo on the container — don't `new SiblingRepository(db)` (server/src/platform/container.ts:99).
- 2026-06-25 — Break a `helpers ↔ repository` (or any cross-module) import cycle by sourcing inferred row TYPES from `db/rows.ts`, not from the owning `repository.ts`. `agents/helpers.ts` importing `AgentRow` from `./repository.js` cycled (repository imports helpers' `isConfigChange`); switching to `../../db/rows.js` broke it. `db/rows.ts` exists precisely so consumers reference a row shape without importing a data layer (server/src/modules/agents/helpers.ts:3).
- 2026-06-21 — Per-run LLM cost is already computed (`outcome.costUsd`, real `usage.cost` for OpenRouter) but `agent_runs.cost_usd` was DROPPED in migration `0009`, so the value was silently discarded in `run-executor`. Re-added the column (`0010`) and threaded `costUsd` → `completeAgentRun` + `trace.stats.cost_usd`. Store `null` (never `0`) for failed/cancelled/unknown so the UI shows "—" not "$0.00" (server/src/modules/reviews/run-executor.ts:213, server/src/modules/reviews/repository/run.repo.ts:141).
- 2026-06-21 — `completeAgentRun`'s param type is declared in TWO places: the thin wrapper class in `repository.ts` AND the impl in `repository/run.repo.ts`. Adding a field to one without the other fails typecheck with a confusing "does not exist in type" at the call site — edit both (server/src/modules/reviews/repository.ts:151).

## Decisions
<!-- decisions with the reason behind them -->
- 2026-07-02 — `seed.ts` writes agents/skills via RAW `db.insert(t.agents / t.skills)`, bypassing `AgentsRepository.insert` / `SkillsRepository.insert`'s transactions entirely (pre-existing convention, confirmed present before this session's Skills-lesson additions). Consequence: the 3 pre-existing built-in agents get NO `agent_versions` row at all (no v1 snapshot), while the 2 new Skills-lesson agents added in this session follow the SAME convention (also no `agent_versions` row) — but the new SKILLS do get an explicit `skill_versions` insert per version (including the 5-entry history for `pr-quality-rubric`), because `SkillsRepository`'s version-snapshot behavior is core to the Versions-tab feature and skipping it would leave that tab empty for seeded data. If an Agent Editor "Versions" tab is ever built, seeded agents will show empty history — that's expected given this asymmetry, not a bug (server/src/db/seed.ts).
- 2026-06-29 — Generalizes the 2026-06-22 cost-invalidation note: because the PR-list row is built from read-time aggregates (`cost_usd`, now `findings`), EVERY mutation that can change them must invalidate the `["pulls"]` query, else the list goes stale with no manual refresh. Now wired in `useDeleteReview`, `useRunReview`, and `useFindingAction` (plus the existing `useDeleteRun`). New list aggregate → audit which mutations touch it and add the `["pulls"]` invalidation (client/src/lib/hooks/reviews.ts).
- 2026-06-25 — ALL feature modules now follow the `routes.ts → service.ts → repository.ts` triple. The 4 previously-"thin" modules (`polling`, `pulls`, `settings`, `workspace`) queried the DB straight from their routes; refactored so routes only parse + delegate, SQL lives in a repository, orchestration in a service. Why: uniform Onion layering, now enforced by a `dependency-cruiser` arch gate (`pnpm arch`). How to apply: mirror `modules/repos`; a new module = the triple + one entry in `modules/index.ts` (server/src/modules/index.ts).
- 2026-06-25 — The PR-list cost-SUM + latest-review-score aggregates moved OUT of `pulls/routes.ts` into `ReviewRepository.totalRunCostByPrIds` / `latestReviewScoreByPrIds` (consumed by `pulls/service.ts`); behaviour unchanged. This SUPERSEDES the `pulls/routes.ts:120` file ref in the 2026-06-22 entry below — the cost-SUM rule still holds, just relocated (server/src/modules/reviews/repository/run.repo.ts, server/src/modules/reviews/repository/review.repo.ts).
- 2026-06-22 — PR list COST column (`PrMeta.cost_usd` from `GET /repos/:id/pulls`) is the SUM of `cost_usd` over ALL completed (`status='done'`) runs for the PR, NOT the latest run's cost (supersedes the 2026-06-21 latest-run choice). It is a read-time aggregate (JS sum over the same `IN` query, no `orderBy`/dedup), so deleting a run from the timeline subtracts its cost on the next fetch — no stored/denormalized total to drift. Skip runs with null/non-finite `cost_usd`; a PR with completed runs but no known cost stays `null` → UI renders "—". Because the total spans runs, `useDeleteRun` must also invalidate the `["pulls"]` query, not just `pr-runs`/`reviews` (server/src/modules/pulls/routes.ts:120, client/src/lib/hooks/reviews.ts:66).

## Recurring Errors & Fixes
<!-- errors that keep coming back + the fix -->
- 2026-07-02 — A field that has BOTH an "import/derived content" size limit AND a persisted Zod write-schema cap will silently dead-end a preview→save flow if the two aren't kept in sync: `MAX_MARKDOWN_BODY_BYTES` (1 MiB, `server/src/modules/skills/constants.ts`) let `POST /skills/import/preview` return a large body, but `SkillWriteInput.body`'s original `.max(80_000)` (copied verbatim from the technical spec, `server/src/vendor/shared/contracts/knowledge.ts:138`) then rejected it on the following `POST /skills` save — found by `/pr-self-review`'s qualitative pass, not by typecheck/tests (both limits are independently valid Zod/const values, so nothing failed until an actual large import was attempted). Fixed by raising the write-schema cap to match. When adding a NEW size/length limit for content that also flows through an import/preview step, grep for every OTHER limit on the same field before picking a number.
